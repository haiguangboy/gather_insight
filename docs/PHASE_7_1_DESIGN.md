# Phase 7.1: High-Value Claim and Evidence Layer

## Scope

Phase 7.1 converts a frozen transcript result into versioned, traceable claim and evidence records. It does not change transcript alignment, repair ASR text, infer exact speakers, write publication copy, or introduce a database.

The workflow supports both established input modes:

- `high_quality_structure_mode`: Result B (`Vecalign raw cosine`) plus meaningful unallocated text and conflicts.
- `dual_text_trend_mode`: the frozen no-uListen dual-text result plus meaningful unmerged text and conflicts.

Both modes use the same intelligence schemas and extraction workflow. Differences in speaker certainty and exact-quotation permission remain explicit data, not hidden heuristics.

## Canonical contract

`prepare-phase71-canonical` reads all three information channels:

1. frozen fused transcript records;
2. meaningful unmerged/unallocated text;
3. conflict records.

It also materializes chapter and speaker metadata. Every canonical record carries a stable `record_id`, time range, source URL, provider, source ranges, speaker status, attribution scope, source hashes, and the input mode. Meaningful unmerged records have the same candidate eligibility as ordinary transcript records, while retaining `from_meaningful_unmerged=true`, `needs_verification=true`, and `exact_quote_allowed=false`.

## Two-pass extraction

### Pass 1: high-recall discovery

The deterministic pass combines:

- topic-block candidates, which preserve enough context for mechanisms and route changes;
- atomic, risk-sensitive sentence candidates, which independently promote numbers, comparisons, causal statements, failure signals, constraints, and technical predictions.

This second path matters because a valuable quantitative sentence can be present in the transcript while being diluted inside a larger topic block. Risks are retained rather than used to discard candidates.

### Pass 2: consolidation and value judgment

The workflow then:

- deduplicates normalized claims within a theme;
- merges evidence and source-record references;
- assigns claim types, value types, themes, scores, and deterministic relations;
- creates trend candidates and a verification queue;
- optionally sends a bounded set of already evidence-bound candidates to the DeepSeek value judge.

The optional judge can classify and score supplied candidates only. It cannot create evidence, rewrite transcript text, repair entities, or add unsupported claims. DeepSeek Flash with thinking is configured through `config/phase_7_1.example.yaml`; `DEEPSEEK_API_KEY` is read only from the environment. Parsed judgments are hash-cached. Runtime API/cache/token counters are returned by the CLI but excluded from canonical reports so cached reruns remain byte-identical.

## Evidence invariants

- `claims.jsonl` may contain a concise claim, but each claim must bind at least one `evidence_id`.
- `evidence.source_text` is copied from canonical transcript or meaningful-unmerged records, never generated by a model.
- Evidence retains `source_record_ids`, ranges, time, provider, URL, and source hashes.
- Exact quotation is permitted only when the source and speaker contract allow it. Section-inferred, boundary-uncertain, unmerged, or risk-bearing material remains reviewable but not directly quotable.
- Numeric, entity, negation, conflict, speaker, and provenance risks produce explicit verification reasons.

## Versioned schemas

The public schemas are:

- `phase_7_1_claim.schema.json`
- `phase_7_1_evidence.schema.json`
- `phase_7_1_theme_assignment.schema.json`
- `phase_7_1_claim_relation.schema.json`
- `phase_7_1_trend_candidate.schema.json`

- `phase_7_1_golden_claim.schema.json`

JSONL remains the fact base. Markdown and HTML are generated views and must not be treated as canonical storage.

## Review and evaluation

`claim_review.html` displays the claim, scores, speaker contract, verbatim evidence, video link, record IDs, source ranges, risks, and meaningful-unmerged provenance. Reviewers can download an independent JSONL file containing `accept`, `reject`, `edit`, or `merge` decisions, verification state, and notes.

The evaluator compares claims with a private 20–30 item golden set and reports recall by category, a precision proxy, meaningful omission, evidence traceability, unsupported claims, speaker-requirement failures, and risk-detection recall. The current YC file is a private reviewer draft selected from the available evidence; it must be reviewed and frozen independently before its scores are treated as a final external benchmark.

```bash
python3 -m gather_insight prepare-phase71-canonical \
  --input-dir <frozen_transcript_output> \
  --manifest <private_manifest.json> \
  --source-input-dir <private_raw_input_dir> \
  --media-root input/media/<media_id>/<mode> \
  --mode dual_text_trend_mode

python3 -m gather_insight extract-phase71 \
  --media-root input/media/<media_id>/<mode> \
  --judge-backend deepseek \
  --judge-config config/phase_7_1.example.yaml \
  --judge-cache-root input/phase_7_1_private_cache

python3 -m gather_insight evaluate-phase71 \
  --golden <private_golden.jsonl> \
  --claims input/media/<media_id>/<mode>/intelligence/claims.jsonl \
  --evidence input/media/<media_id>/<mode>/intelligence/evidence.jsonl \
  --output input/media/<media_id>/<mode>/reports/golden_evaluation.json
```
